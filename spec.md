# The unPaper HTML Specification

**Version 1**

A profile of HTML that is, at the same time, a print-ready slide deck viewable in
any browser and a faithful, machine-readable description of an editable PowerPoint
(`.pptx`) file. A document that conforms to this specification converts to native
PowerPoint objects predictably — by mechanical lowering, not by guesswork.

---

## Status of this document

This is **Version 1** of the unPaper HTML specification. It is a published, openly
implementable format specification: anyone **MAY** build software that reads or
writes unPaper HTML, for any purpose, including commercial purposes, without
permission or fee.

The specification is edited and stewarded by **reHeritage GmbH** (Berlin,
Germany). It is versioned; the conformance levels defined here are stable, so a
document that conforms to a given version of this specification continues to
convert predictably under that version.

The *unPaper converter* — software that compiles unPaper HTML into PowerPoint — is
a separate product of reHeritage GmbH, governed by its own terms, and is **not**
covered by this specification or its licence. This document describes the format,
not any one implementation of it.

## 1. Conventions

The key words **MUST**, **MUST NOT**, **REQUIRED**, **SHALL**, **SHALL NOT**,
**SHOULD**, **SHOULD NOT**, **RECOMMENDED**, **MAY**, and **OPTIONAL** in this
document are to be interpreted as described in [RFC 2119] and [RFC 8174] when, and
only when, they appear in all capitals, as shown here.

Code shown in `monospace` is normative where it illustrates a required structure
and informative where it illustrates a suggestion.

## 2. Design rationale (informative)

Every other HTML-to-PowerPoint tool treats the HTML as material found in the wild
and attempts to recover meaning from arbitrary markup. unPaper HTML inverts this:
the document is *written to a specification*, so a conforming converter does not
guess — it lowers a known intermediate representation into native PowerPoint
objects. The richer the contract a document honours, the more of it arrives as
editable native objects rather than as a flat picture.

The format is designed to be generated by software (typically an AI assistant),
which makes it practical to demand a precision and redundancy that would be
tedious for a human author. Nothing in it, however, is specific to any one
generator or converter.

## 3. Conformance levels

A document's guarantees depend on how much of the contract it follows. A conforming
document **MUST** satisfy at least Level 0; higher levels are cumulative.

| Level | Name | Guarantee |
| --- | --- | --- |
| **L0** | Any fixed-size deck | Best-effort conversion. A file is always produced; how much is editable varies. |
| **L1** | Geometry contract | Fixed-size, self-contained, renders without scripts (§4). Pixel-faithful conversion. |
| **L2** | Semantic contract | Uses elements for their meaning (§5). Text, lists, tables and shapes arrive native and editable; the deck's palette informs the theme. |
| **L3** | Data contract | Data visuals carry their data (§6). Chart numbers are editable in PowerPoint; the deck is regenerable from its own description. |

A converter **SHOULD** detect and report the level it determined for each document,
and **MAY** report per-slide which level each element reached.

A document **MAY** declare the highest level it targets; see §8.

## 4. The geometry contract (L1)

A conforming document at Level 1 or above:

1. **MUST** be a single, self-contained HTML file. It **MUST NOT** reference
   external CSS, JavaScript, fonts, or images, and **MUST NOT** make network
   requests. Every visual **MUST** be expressed as inline SVG, CSS, or a `data:`
   URI.
2. **MUST** render completely with JavaScript disabled. Scripts **MAY** be present
   for progressive enhancement on screen, but no content (text, numbers, images,
   page chrome) may depend on them, because a converter reads the document with
   scripting disabled.
3. **MUST** represent each slide as a single element carrying the class `slide`,
   sized to a fixed canvas. The **RECOMMENDED** canvas is 1280×720 CSS pixels
   (PowerPoint 16:9). A document **MAY** use another fixed size; all slides in one
   document **SHOULD** share the same size.
4. **SHOULD** define an `@page` rule matching the slide size with zero margin, so
   that printing yields one page per slide.

```html
<section class="slide">
  <h1>One claim per slide, written as a full sentence</h1>
  <p>Supporting detail…</p>
</section>
```

Line breaks as laid out by the browser are authoritative: a converter bakes them in
and disables re-wrapping, so a missing font can never reflow the slide.

## 5. The semantic contract (L2)

At Level 2, an author **MUST** express meaning with the element that carries it, so
each converts to the corresponding native PowerPoint object:

- A data or comparison grid **MUST** be a real `<table>` (`<tr>`/`<th>`/`<td>`,
  with `colspan`/`rowspan` where needed). It **MUST NOT** be simulated with
  `<div>`s. → a native, editable table.
- A list **MUST** use `<ul>`/`<ol>` with `<li>`. → real bullet paragraphs.
- A link **MUST** use `<a href>` with an absolute URL. → a live hyperlink.

Cards, panels, dividers and accent rules built with ordinary CSS (background,
border, `border-radius`, gradient, `box-shadow`) are reconstructed as movable
native shapes; inline `<svg>` icons are reconstructed as vector pictures or, where
the geometry permits, native freeform shapes. Authors **SHOULD** keep one idea per
slide and **SHOULD NOT** rely on visual tricks that have no native equivalent
(see §7).

## 6. The data contract (L3)

A visual whose data the author wants to remain editable **MUST** carry that data as
a `data-unpaper-*` attribute, in addition to its inline rendering.

### 6.1 Charts — `data-unpaper-chart`

Draw the chart as inline SVG for the on-screen and print rendering, and attach its
data, on the `<svg>` or a wrapping element, as a JSON object:

```html
<div data-unpaper-chart='{
  "type": "bar",
  "categories": ["Q1","Q2","Q3","Q4"],
  "series": [
    { "name": "Revenue", "values": [4.2,5.1,4.8,6.3], "color": "F0B429" }
  ],
  "axis": { "min": 0, "max": 8 }
}'>
  <svg viewBox="0 0 760 320"> … </svg>
</div>
```

- `type` **MUST** be one of `bar`, `column`, `line`, `area`, `pie`, `doughnut`.
- `categories` **MUST** be a non-empty array; each series' `values` **MUST** have
  the same length as `categories` and contain only finite numbers.
- `color` values, where given, **MUST** be six-digit hex without a leading `#`.
- `axis.min`/`axis.max`, where given, **SHOULD** match the scale the SVG actually
  draws, so the native chart and the SVG agree.
- Numbers a reader must see (for example, a percentage in a doughnut's centre)
  **MUST** be HTML positioned over the SVG, **NOT** SVG `<text>`, because the
  converter replaces the SVG with the native chart.

A converter that finds a valid `data-unpaper-chart` **MUST** replace the inner SVG
with a native, data-editable chart at the SVG's position. On invalid or missing
data it **MUST** fall back to rendering the SVG as a picture (L2 behaviour).

### 6.2 Speaker notes — `data-unpaper-notes`

Per-slide talking points **SHOULD** be placed in an `<aside class="notes">` (or any
element carrying `data-unpaper-notes`) before the slide closes. Such an element
**MUST** be hidden from the rendered slide and becomes the slide's native speaker
notes.

## 7. Security and privacy considerations (informative)

Because a conforming document is self-contained and makes no network requests
(§4.1), it can be authored, viewed, printed and converted entirely on one machine,
with no data leaving it. This is a deliberate property: it keeps confidential
content local. Authors and implementers **SHOULD** preserve it — in particular,
converters **SHOULD** operate without uploading the document.

## 8. Versioning

A document **MAY** declare the specification version it targets on its root
element, so it keeps converting predictably as the specification grows:

```html
<html data-unpaper="1">
```

All of this format's semantic attributes share the `data-unpaper-` prefix. (An
earlier prefix, `data-h2p-`, predates this name; converters **MAY** continue to
accept it, but documents **SHOULD** use `data-unpaper-`.)
